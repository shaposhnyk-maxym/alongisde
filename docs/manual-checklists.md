# Manual checklists

Деякі речі свідомо не автоматизуються в CI (системні UI-флоу, реальні
пристрої, живі push-сервіси) — див. відповідні мілстоуни в
`docs/roadmap.md`. Кожен такий пункт проганяється вручну хоча б раз
перед мержем PR і фіксується тут з датою й результатом.

---

## M5 — Login (Google Sign-In)

**Чому вручну:** сам виклик нативного SDK (Credential Manager на
Android, GIDSignIn на iOS) — системний UI-флоу з реальним
Google-акаунтом, його не можна надійно автоматизувати в CI. Юніт- та
інтеграційні тести покривають усе навколо цього виклику (`AuthContainer`,
обмін токена на сесію) — сам виклик перевіряється тільки тут.

### Android (реальний пристрій)

- [ ] Натискання "Continue with Google" відкриває системний Credential
      Manager account picker
- [ ] Вибір акаунта успішно завершує sign-in: екран переходить у
      сигнальний стан (`session` заповнений), без помилки
- [ ] Скасування пікера (back / тап поза діалогом) повертає екран у
      idle-стан без показу помилки
- [ ] Airplane mode увімкнено після вибору акаунта (симуляція мережевого
      збою під час обміну токена) → показується network-помилка,
      кнопка знову доступна для повтору
- [ ] Після успішного sign-in: force-stop застосунку (Settings → Apps →
      Force stop), відкрити знову → одразу залогинений, без пікера,
      без мережевого запиту (кешована сесія ще не протермінована)
- [ ] Ізолювати кейс протермінованої сесії (напр. тимчасово зменшити
      safety margin/`expiresInSeconds` локально або почекати годину) →
      перезапуск застосунку тихо ре-авторизується через Credential
      Manager без показу пікера (лог `GoogleAuthProvider: signIn:
      requesting credential (filterByAuthorizedAccounts=true)` +
      `signIn: success`), користувач одразу бачить залогинений стан

### iOS

- [ ] **BLOCKED** — `iosApp` (Xcode-проєкт) ще не існує, Apple
      Developer акаунт заблокований (див. пам'ять сесії). `GoogleAuthProvider`
      спроєктований під Swift-реалізацію (callback-based, без suspend),
      але сам GIDSignIn-код і мануальна перевірка відкладені до моменту,
      коли iOS-таргет реально з'явиться в проєкті. Зафіксовано як
      свідоме відхилення від Accept-критерію M5, а не пропуск.

## M10 — Diary: capture & processing pipeline

**Чому вручну:** M10's Accept-критерії не вимагають мануальної
перевірки (на відміну від M5/M7/M13/M17) — увесь пайплайн покритий
юніт-тестами на синтетичних даних. Цей пункт — додатковий, не з
Accept-критеріїв: реальний наскрізний прогін (справжні GPS-фото →
кластеризація → Google Geocoding → Gemini-опис) вимагає живих API-ключів
і зараз не автоматизується (real API calls, платні квоти).

**Тестові фото підготовлені** (2026-07-19): 9 реальних geotagged JPEG
(NIKON COOLPIX, з `github.com/ianare/exif-samples`, jpg/gps/) —
5 фото формують "епізод 1" (реальні EXIF, ~16 хв, макс 274м між сусідніми
кадрами), 4 фото — синтетично зсунуті на +4 год / +~4.3км GPS відносно
оригіналу, формуючи чітко окремий "епізод 2" (перевірено: обидва пороги
кластеризації — 2год і 500м — перетнуті між епізодами, всередині кожного
епізоду — ні). Запушені на фізичний пристрій (`adb push` →
`/sdcard/DCIM/AlongsideTest/`) + `MEDIA_SCANNER_SCAN_FILE`-broadcast;
підтверджено через `content query` на `content://media/external/images/media`
— усі 9 проіндексовані, `datetaken` збігається з очікуваним EXIF.

- [x] **Google Geocoding API** (2026-07-19): реальний виклик через `curl`
      з координатами обох тестових епізодів, ключ з `local.properties`.
      `status: OK` обидва рази, JSON-форма 1:1 збігається з нашими DTO.
      Виявлено й виправлено: `preferredPlaceName()` без `route`/
      `neighborhood` у пріоритеті давав однакову назву для різних вулиць
      того самого міста (див. roadmap.md).
- [x] **Gemini vision-опис** (2026-07-19): реальний виклик з 4 представницькими
      фото епізоду 1 (base64, той самий промпт-формат, що
      `GeminiVisionDescriptionClient`) → `gemini-flash-latest`. Отримано
      справжній емоційний абзац, що відповідає концепт-документу.
      Виявлено й виправлено: дефолтна модель `gemini-2.0-flash` мала
      нульову free-tier квоту; error-парсинг губив реальні
      billing-повідомлення (див. roadmap.md — обидва фікси окремими
      комітами).
- [ ] **Не пройдено ще**: той самий виклик, але через РЕАЛЬНИЙ код
      (`EpisodeProcessingPipeline` + `AndroidExifPhotoReader` +
      `AndroidPhotoByteReader` на пристрої, не `curl`/Python напряму) —
      заблоковано відсутністю UI/entry point у M10 (свідомо, Timeline
      UI — M12) і відсутністю запиту `ACCESS_MEDIA_LOCATION` дозволу
      (див. roadmap.md). Потрібен тимчасовий debug-тригер (напр. кнопка
      в Debug Menu з M19) — не існує в кодовій базі, зробити окремим
      кроком. `curl`/Python-перевірка вище підтверджує, що API-виклики й
      формати запитів/відповідей коректні — лишається перевірити тільки
      android-специфічну частину (EXIF-читання з permission, byte-читання
      фото).
- [ ] Перевірити, що `AndroidExifPhotoReader` реально повертає GPS з цих
      фото на пристрої — наразі неможливо: `ACCESS_MEDIA_LOCATION`
      задекларовано в маніфесті, але runtime-запит permission ще не
      заведений (див. `docs/roadmap.md` M10), тож дозвіл не надано і
      `exif.latLong` буде `null` навіть із цими фото на пристрої.

## M12 — Timeline: UI & animations

**Чому вручну:** M12's Accept-критерії не вимагають мануальної
перевірки (усе покрито screenshot/Compose UI/юніт-тестами), але живий
пристрій — єдиний спосіб відчути реальну плавність свайпу/анімацій,
на відміну від Robolectric. У цьому середовищі розробки немає
доступного `adb`/емулятора, тож нижче не пройдено жодного пункту.

- [ ] Реальний свайп по каруселі на emulator/пристрої: лічильник → День
      1 → ... → День N, плавно, без ривків/пропусків сторінок
- [ ] Заблокований день на реальному пристрої дійсно показує пояснення
      стану очікування (не просто замок) для всіх трьох варіантів
- [ ] `./gradlew :androidApp:assembleDebug` встановлюється й
      запускається на emulator (Pixel 8 Pro, API 36 — той самий, що
      M6/M8 використовували) без крашу при вході на вкладку Timeline

## M13.2 — Places: share-link entry point

**Чому вручну:** Android-частина (intent-filter, `PlaceImportContainer`,
screenshot-тести) повністю покрита автоматизованими тестами. iOS Share
Extension — системний UI-флоу (той самий клас проблеми, що GIDSignIn у
M5), не автоматизується в CI.

### Android (реальний пристрій/emulator)

- [ ] У Google Maps: "Share" на реальному місці → "Alongside" з'являється
      в списку застосунків-отримувачів
- [ ] Вибір "Alongside" відкриває картку підтвердження з реальною назвою
      місця, фото й рейтингом (не placeholder-стан)
- [ ] "Add to Places" зберігає місце — видиме через
      `PlaceCandidateRepository.observeByTrip` (перевірити логами/дебагером,
      поки немає реального Places-екрана — M16)
- [ ] Другий "Share" одразу після першого (застосунок ще відкритий) —
      `onNewIntent` спрацьовує, з'являється нова картка підтвердження,
      а не крах/зависання
- [ ] Airplane mode увімкнено одразу після "Add to Places" (симуляція
      короткого зникнення мережі під час фото-завантаження) → після
      повернення мережі й відкриття вкладки Places лог показує, що
      `PlaceRetryDataSource` дозаливає фото, що не встигли (30с
      poll-інтервал)

### iOS

- [ ] **BLOCKED** — `iosApp` (Xcode-проєкт) ще не існує, той самий
      Apple Developer account issue, що M5/M7/M10. Share Extension
      спроєктований лише на рівні Accept-критерію M13.2 (мануальний
      чекліст), сам код і перевірка відкладені до появи реального
      iOS-таргету.
