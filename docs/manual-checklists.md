# Manual checklists

Деякі речі свідомо не автоматизуються в CI (системні UI-флоу, реальні
пристрої, живі push-сервіси) — див. відповідні мілстоуни в
`docs/roadmap.md`. Кожен такий пункт проганяється вручну хоча б раз
перед мержем PR і фіксується тут з датою й результатом.

---

## M5 — Login (Google Sign-In)

**Чому вручну:** сам виклик нативного SDK (Credential Manager на
Android, GIDSignIn на iOS) — системний UI-флоу з реальним
Google-акаунтом, його не можна надійно автоматизувати в CI. Юніт- та
інтеграційні тести покривають усе навколо цього виклику (`AuthContainer`,
обмін токена на сесію) — сам виклик перевіряється тільки тут.

### Android (реальний пристрій)

- [ ] Натискання "Continue with Google" відкриває системний Credential
      Manager account picker
- [ ] Вибір акаунта успішно завершує sign-in: екран переходить у
      сигнальний стан (`session` заповнений), без помилки
- [ ] Скасування пікера (back / тап поза діалогом) повертає екран у
      idle-стан без показу помилки
- [ ] Airplane mode увімкнено після вибору акаунта (симуляція мережевого
      збою під час обміну токена) → показується network-помилка,
      кнопка знову доступна для повтору
- [ ] Після успішного sign-in: force-stop застосунку (Settings → Apps →
      Force stop), відкрити знову → одразу залогинений, без пікера,
      без мережевого запиту (кешована сесія ще не протермінована)
- [ ] Ізолювати кейс протермінованої сесії (напр. тимчасово зменшити
      safety margin/`expiresInSeconds` локально або почекати годину) →
      перезапуск застосунку тихо ре-авторизується через Credential
      Manager без показу пікера (лог `GoogleAuthProvider: signIn:
      requesting credential (filterByAuthorizedAccounts=true)` +
      `signIn: success`), користувач одразу бачить залогинений стан

### iOS

- [ ] **BLOCKED** — `iosApp` (Xcode-проєкт) ще не існує, Apple
      Developer акаунт заблокований (див. пам'ять сесії). `GoogleAuthProvider`
      спроєктований під Swift-реалізацію (callback-based, без suspend),
      але сам GIDSignIn-код і мануальна перевірка відкладені до моменту,
      коли iOS-таргет реально з'явиться в проєкті. Зафіксовано як
      свідоме відхилення від Accept-критерію M5, а не пропуск.

## M10 — Diary: capture & processing pipeline

**Чому вручну:** M10's Accept-критерії не вимагають мануальної
перевірки (на відміну від M5/M7/M13/M17) — увесь пайплайн покритий
юніт-тестами на синтетичних даних. Цей пункт — додатковий, не з
Accept-критеріїв: реальний наскрізний прогін (справжні GPS-фото →
кластеризація → Google Geocoding → Gemini-опис) вимагає живих API-ключів
і зараз не автоматизується (real API calls, платні квоти).

**Тестові фото підготовлені** (2026-07-19): 9 реальних geotagged JPEG
(NIKON COOLPIX, з `github.com/ianare/exif-samples`, jpg/gps/) —
5 фото формують "епізод 1" (реальні EXIF, ~16 хв, макс 274м між сусідніми
кадрами), 4 фото — синтетично зсунуті на +4 год / +~4.3км GPS відносно
оригіналу, формуючи чітко окремий "епізод 2" (перевірено: обидва пороги
кластеризації — 2год і 500м — перетнуті між епізодами, всередині кожного
епізоду — ні). Запушені на фізичний пристрій (`adb push` →
`/sdcard/DCIM/AlongsideTest/`) + `MEDIA_SCANNER_SCAN_FILE`-broadcast;
підтверджено через `content query` на `content://media/external/images/media`
— усі 9 проіндексовані, `datetaken` збігається з очікуваним EXIF.

- [ ] **Не пройдено ще**: реальний виклик `EpisodeProcessingPipeline`
      проти цих фото з живими `GOOGLE_PLACES_API_KEY`/`GEMINI_API_KEY`
      (див. `docs/local-setup.md`) — заблоковано відсутністю UI/entry
      point у M10 (свідомо, Timeline UI — M12). Потрібен тимчасовий
      debug-тригер (напр. кнопка в Debug Menu з M19, або ізольований
      JVM-скрипт, що читає ці 9 фото напряму з диска, оминаючи
      `ExifPhotoReader`/`ContentResolver`, і б'є прямо в реальні
      `GooglePlacesGeocodingClient`/`GeminiVisionDescriptionClient`) —
      не існує в кодовій базі, зробити окремим кроком.
- [ ] Перевірити, що `AndroidExifPhotoReader` реально повертає GPS з цих
      фото на пристрої — наразі неможливо: `ACCESS_MEDIA_LOCATION`
      задекларовано в маніфесті, але runtime-запит permission ще не
      заведений (див. `docs/roadmap.md` M10), тож дозвіл не надано і
      `exif.latLong` буде `null` навіть із цими фото на пристрої.
