# Бекенд — план для новачка

## Головне рішення
Не пишемо традиційний сервер. Бекенд-інфраструктура = **Firebase**
(Firestore + Auth + FCM). Клієнт звертається до неї через **Ktor +
REST API** (не через multiplatform Firebase SDK — той має проблеми з
CocoaPods-лінкуванням на iOS).

## Що не потребує коду взагалі
- **Хто що бачить, owner-only delete** — описується Firestore Security
  Rules (декларативний файл правил доступу, не код застосунку)
- **Інвайт-код** — генерується на клієнті, унікальність перевіряється
  звичайним запитом до Firestore

## Що потребує кількох рядків серверного коду
Тільки одна річ: тригер "обидва дні готові → пуш обом одночасно".

Це одна Cloud Function (JS/TypeScript), яка:
1. Спрацьовує автоматично при записі нового DiaryEntry в Firestore
2. Перевіряє, чи в обох учасників поїздки є DiaryEntry на цю дату
   зі статусом synced
3. Якщо так — викликає Firebase Admin SDK, щоб надіслати FCM push
   обом учасникам

Приблизно 15-20 рядків коду, Firebase Console дає майстер налаштування
проєкту під Cloud Functions.

## Практичний нюанс
Cloud Functions вимагають увімкнути план Blaze (платний), навіть попри
щедрий безкоштовний ліміт (2 млн викликів/місяць, 50К read/день у
Firestore). Для двох користувачів на тиждень витрати гарантовано $0 —
просто постав budget alert у Firebase Console, щоб бачити, якщо щось
піде не так (нескінченний цикл у функції тощо).

## Порядок дій
1. Створити Firebase-проєкт, увімкнути Auth (Google), Firestore,
   Cloud Messaging
2. Написати Firestore Security Rules (owner-only delete, member-only
   read/write в межах своєї поїздки)
3. Написати 1 Cloud Function для symmetric-unlock push тригера
4. З клієнта (Ktor) звертатись до Firestore через REST API —
   звичайні HTTP-запити, як до будь-якого іншого API
