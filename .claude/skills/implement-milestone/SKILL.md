# Implement Milestone: $ARGUMENTS

## Контекст, який треба прочитати першим
1. `CLAUDE.md` — стек, архітектура, конвенції, TDD-підхід, git-флоу
2. `docs/roadmap.md` — знайди мілстоун $ARGUMENTS, зверни особливу увагу
   на його Accept-критерії — вони й визначають, що саме "готово"
3. `docs/kmp-module-architecture.md` — якщо мілстоун торкається модуля,
   структура якого потребує уточнення

## Процес
1. Створи гілку `feat/<milestone-id>-<коротка-назва>` від `main`
   (наприклад `feat/m8-pairing`)
2. Дотримуйся TDD: спочатку тест (юніт / інтеграційний / screenshot —
   залежно від того, чого вимагає Accept-критерій цього мілстоуна), тест
   має впасти, потім реалізація, потім тест зелений
3. Реалізуй ТІЛЬКИ те, що описано в Accept-критеріях цього мілстоуна —
   не забігай наперед на наступні
4. Дотримуйся конвенцій коду з CLAUDE.md: Google Kotlin Style Guide,
   MVI-найменування (`State`/`Intent`/`SideEffect`/`Container`),
   модульні межі (feature-модулі не залежать один від одного напряму,
   `core:model`/`core:domain` — без жодного expect/actual)
5. Перед комітом прожени локально: `./gradlew ktlintCheck detekt
   allTests` — усе має бути зелене
6. Коміти — Conventional Commits, узгоджені з префіксом гілки
   (`feat:`, `test:`, `refactor:` тощо)
7. Після завершення онови `docs/roadmap.md`: познач мілстоун як
   виконаний, додай короткий нотейт, якщо було відхилення від
   початкового плану і чому

## Якщо Accept-критерій незрозумілий або суперечить наявному коду
Зупинись і опиши знайдену суперечність користувачу замість того, щоб
вгадувати — дешевше виправити зараз, ніж після реалізації.

## Перед тим, як вважати мілстоун завершеним
Пройдись по кожному пункту Accept-критеріїв $ARGUMENTS з roadmap.md
явним чек-листом і підтверди кожен окремо — не одним загальним "готово".
